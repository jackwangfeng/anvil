# bianyi —— 用 Rust 实现的 C 编译器 · 设计文档

- 日期：2026-05-30
- 状态：已通过头脑风暴，待用户最终评审
- 作者：协作设计（用户 + Claude）

## 1. 目标与定位

用 **Rust 从头手写**一个尽量完整的 **C 编译器**（C89/C99 方向），后端**直接生成 AArch64 汇编**，再调用系统 `clang` 汇编链接成原生可执行文件。

定位是**硬核 / 学习导向**：最大化对编译器全流程的理解，因此：

- 不依赖任何第三方编译相关 crate（不用 parser 生成器、不用 LLVM 绑定）。
- 预处理器、词法、语法、语义、IR、代码生成、寄存器/栈帧，全部手写。

非目标（明确排除，避免范围失控）：

- 不追求与 GCC/Clang 的优化水平对齐。
- 不追求 100% 覆盖 C 标准的每个角落特性。
- M6 之后的特性（见 §3）属"未来工作"，本项目不承诺交付。

## 2. 架构：七阶段编译流水线

数据沿一条单向管道流动，每阶段职责单一、接口清晰、可独立测试。

```
.c 源文件
   │
   ▼ 1. 预处理器 (Preprocess)   #include / #define / 宏展开 / 条件编译 → 展开后 token 流
   ▼ 2. 词法分析 (Lexer)        → token 流（关键字/标识符/常量/字符串/运算符/标点，带位置）
   ▼ 3. 语法分析 (Parser)       递归下降 + 表达式 Pratt 优先级 → AST
   ▼ 4. 语义分析 (Semantic)     符号表/作用域/类型检查/隐式转换/左值检查 → typed AST
   ▼ 5. 中间代码 (IR Gen)       → 线性三地址码风格 IR
   ▼ 6. 代码生成 (Codegen)      IR → AArch64 汇编文本（调用约定/栈帧/寄存器）
   ▼ 7. 汇编链接                调用系统 clang 把 .s 汇编+链接 → 可执行文件
   │
   ▼ 可执行文件
```

### 关键设计决策

1. **保留独立 IR 层（第 5 步）**。多花前期成本，换取代码生成器的清晰，并为以后做优化、甚至换后端（如 x86）留接口。不采用 AST 直接生成汇编。
2. **报错质量从第一天起**。每个 token 携带源码位置（行/列），词法/语法/语义阶段统一产出 `file:line:col: error` 风格报错。`span.rs` + `error.rs` 是贯穿全程的基础设施。
3. **核心契约优先**。`ast.rs`、`types.rs`、`ir/` 三个数据结构是所有阶段共享的契约，需先稳定其定义再大规模填充各阶段逻辑。

## 3. 里程碑拆分

C 太大，分 7 个里程碑递进。**每个里程碑结束时都有一个能编译真实 C 程序、能产出正确可执行文件的编译器**，支持的 C 子集逐步变大。可在任意里程碑停下。

| # | 里程碑 | 跑通后能编译 | 核心新增能力 |
|---|--------|------------|------------|
| **M0** | 端到端骨架 | `int main(){ return 42; }` | 搭起七阶段管道，跑通 lexer→parser→IR→AArch64→clang→可执行文件。**暂时跳过预处理器**（直接吃 .c），用退出码验证整链路 |
| **M1** | 表达式与整数运算 | `return 1+2*3-(4/2);` | 整型常量、四则、取模、括号、一元正负；Pratt 优先级；IR 三地址码；栈式求值 |
| **M2** | 变量与语句 | 局部变量、赋值、`if/else`、`while`、`for`、`{}` 块、比较/逻辑/位运算 | 符号表、作用域、栈帧布局、控制流（标签+跳转汇编） |
| **M3** | 函数 | 多函数、传参、返回值、递归（如 `fib`） | AArch64 调用约定（x0–x7 传参、栈传参、栈帧保存/恢复）、函数声明/定义 |
| **M4** | 类型系统与指针 | `int*`、`char`、数组、`&`/`*`、指针运算、`sizeof`、隐式转换、字符串字面量 | 完整标量类型、指针/数组语义、左值规则、类型大小与对齐 |
| **M5** | 聚合类型 | `struct`、`union`、`enum`、`typedef`、`.`/`->`、嵌套 | 复合类型布局、成员偏移、字段对齐 |
| **M6** | 自研预处理器 | `#include`、`#define`（对象式+函数式宏）、`#if/#ifdef/#endif`、`#`/`##` | 独立预处理子系统，接到管道最前端，替换 M0 的"直接吃 .c" |

### 顺序与依赖

- **先做编译器主体（M0–M5），最后做自研预处理器（M6）**。预处理器是相对独立的子系统，放最后不阻塞主线，且届时对 token 流已熟悉。
- M0–M5 期间，测试程序**只用不依赖标准库头文件的纯计算程序**，用退出码（必要时 stdout）验证，不依赖 `printf`/`#include`。
- 里程碑之间是硬依赖链（M3 依赖 M1+M2，codegen 依赖 IR 依赖 parser……），因此整体串行推进。

### 未来工作（M6 之后，本项目不承诺）

可变参数函数、函数指针、`switch`、`goto`、`static`/`extern`、初始化器列表、与 libc 链接调用 `printf` 等。

## 4. 执行策略（多 agent 协作）

**串行为主（方案一）**：一个里程碑一个里程碑做，因为编译器各阶段是硬依赖链，"按里程碑切给多 agent 并行"会因共享契约（AST/IR/类型）导致严重集成冲突，是陷阱。

并行只用在**真正独立的子任务**上，例如：

- 一个 agent 写实现的同时，另一个 agent 扩充 `tests/cases/` 用例库。
- M4/M5 类型系统：一个 agent 做类型表示，另一个做布局/对齐，接口先定好。
- M6 预处理器是独立子系统，可在主线收尾阶段并行开发。

不采用"15 agent 齐冲"。并行用在刀刃上。

## 5. 技术选型与项目结构

- 纯 Rust，无第三方编译相关 crate。
- 单 Cargo crate：`bianyi` 既是库（各阶段模块）又是 CLI（`bianyi foo.c -o foo`）。

```
bianyi/
├── Cargo.toml
├── src/
│   ├── main.rs          # CLI 入口：解析参数，串管道，调 clang 汇编链接
│   ├── lib.rs           # 各阶段串成 compile() 管道
│   ├── span.rs          # 源码位置 (行/列)，报错基础
│   ├── error.rs         # 统一 file:line:col 报错类型
│   ├── preprocess/      # M6：自研预处理器
│   ├── lexer/           # 词法分析
│   ├── parser/          # 语法分析 → AST
│   ├── ast.rs           # AST 定义（核心契约）
│   ├── sema/            # 语义分析 → typed AST
│   ├── types.rs         # 类型系统（核心契约）
│   ├── ir/              # IR 定义 + 生成（核心契约）
│   └── codegen/         # IR → AArch64 汇编
├── tests/
│   ├── cases/           # .c 用例库，每个含期望退出码（同名 .expected 或约定）
│   └── integration.rs   # 编译并运行每个 case，比对退出码/输出
└── docs/superpowers/specs/   # 本设计文档
```

约束（来自项目 CLAUDE.md）：单文件 < 500 行；不往根目录扔工作文件；改完跑测试；构建通过再提交。

## 6. 测试与验证（集成测试为主，方案 B）

- `tests/cases/*.c`：不断壮大的真实 C 程序库，每个标注期望退出码（和/或 stdout）。
- 集成测试对每个 case 跑完整 `编译 → clang 链接 → 执行`，比对结果。
- 每个里程碑结束时，"通过几个 case" 即进度的直观度量。
- 少量单元测试覆盖棘手处：宏展开、寄存器/栈帧分配、类型对齐。

## 7. 环境前提

- 目标机器：macOS / AArch64（Apple Silicon）。
- 依赖系统 `clang` 做汇编链接（已随 Xcode CLT 提供）。
- **正式开工前需安装 Rust 工具链**（本机当前无 `rustc`/`cargo`）：通过 `rustup` 安装。
